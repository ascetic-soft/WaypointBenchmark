# Бенчмарк роутера Waypoint

Набор тестов для сравнения роутера [Waypoint](https://github.com/ascetic-soft/Waypoint) с популярными PHP-библиотеками маршрутизации.

## Тестируемые роутеры

|--------|-------|--------|----------|
| **Waypoint** | [ascetic-soft/waypoint](https://github.com/ascetic-soft/Waypoint) | v1.2.1 | PSR-15 роутер с prefix-trie матчингом |
| **FastRoute** | [nikic/fast-route](https://github.com/nikic/FastRoute) | v1.3.0 | Популярный standalone regex-роутер |
| **Symfony** | [symfony/routing](https://symfony.com/doc/current/routing.html) | v7.4.4 | Компонент маршрутизации Symfony |
| **Laravel** | [illuminate/routing](https://github.com/illuminate/routing) | v11.47.0 | Роутер фреймворка Laravel |
| **League** | [league/route](https://github.com/thephpleague/route) | v6.2.0 | PSR-15 совместимый роутер |
| **Nette** | [nette/routing](https://github.com/nette/routing) | v3.1.2 | Роутер фреймворка Nette |

## Требования

- PHP >= 8.4
- ext-mbstring
- Composer

## Установка

```bash
git clone https://github.com/ascetic-soft/Waypoint-benchmark.git
cd Waypoint-benchmark
composer install
```

## Использование

### CLI-бенчмарк (быстрое сравнение)

Запустить все сценарии со всеми роутерами:

```bash
php bin/benchmark
```

Фильтр по роутеру:

```bash
php bin/benchmark --router=waypoint,fastroute,symfony
```

Фильтр по сценарию:

```bash
php bin/benchmark --scenario=static,dynamic
```

Доступные сценарии: `static`, `dynamic`, `highload`

Изменить количество запусков (по умолчанию 20):

```bash
php bin/benchmark --runs=50
```

### PHPBench (статистические бенчмарки)

Запустить все бенчмарки:

```bash
vendor/bin/phpbench run --report=aggregate
```

Фильтр по группе:

```bash
vendor/bin/phpbench run --report=aggregate --group=static
vendor/bin/phpbench run --report=aggregate --group=dynamic
vendor/bin/phpbench run --report=aggregate --group=highload
```

### Makefile

```bash
make install      # composer install
make bench        # Запуск CLI-бенчмарка
make phpbench     # Запуск PHPBench
make all          # Установка + бенчмарк
```

## Методология

> **Каждый бенчмарк измеряет реальный цикл запрос-ответ с раздельным замером прогрева и рабочего запроса.**
>
> Каждый «запрос» = полная инициализация роутера + диспетчеризация:
> - **Кэшируемые роутеры** (FastRoute, Symfony, Waypoint): кэш прогревается один раз, затем каждый запрос загружает данные из кэша + диспетчеризирует
> - **Некэшируемые роутеры**: каждый запрос инициализирует роутер с нуля + регистрирует все маршруты + диспетчеризирует
>
> Замер времени разделён на две фазы:
> - **Прогрев (Warmup)** — первый (холодный) запрос после построения кэша. Может включать промахи opcache, накладные расходы автозагрузчика и т.д.
> - **Запрос (Request)** — медиана последующих (тёплых) запросов. Отражает стабильную производительность в продакшене.
>
> Все 9 роутеров соревнуются в каждом сценарии на равных условиях. Роутеры без поддержки кэширования платят полную стоимость инициализации — это их компромисс.

## Анализ и выводы

### Классификация по уровням

На основе стабильного времени запроса (после прогрева), где кэшируемые роутеры используют свой кэш:

**Уровень 1 — Самая быстрая обработка запросов** (скомпилированные/кэшированные диспетчеры):
- **Waypoint** — матчинг на основе prefix-trie с поддержкой кэша. **1-е место в 4 из 9 сценариев**, включая все диспетчеризации «последнего маршрута» и все тесты на масштабе (500+ маршрутов). При 1 000 смешанных маршрутах Waypoint показывает 0.683 мс — **в 2.3 раза быстрее FastRoute**. В данном бенчмарке Waypoint использует свой trie-матчер напрямую (без PSR-15 middleware-пайплайна), так же как FastRoute и Symfony используют свои базовые матчеры.
- **FastRoute** — самый быстрый в 5 из 9 сценариев (диспетчеризация первого маршрута и малый/средний масштаб). `cachedDispatcher()` загружает скомпилированные regex-данные из PHP-файла, обеспечивая 0.001 мс на одиночную диспетчеризацию при 100 статических маршрутах.

**Уровень 2 — Кэшированный средний класс** (хорош для одиночных запросов, отстаёт на масштабе):
- **Symfony** — `CompiledUrlMatcher` хорошо показывает себя при одиночных диспетчеризациях (2-е место в динамическом первом маршруте — 0.002 мс), но уходит на далёкое 3-е место во всех сценариях с множественными запросами. При 1 000 маршрутах в 57 раз медленнее Waypoint (39.172 мс против 0.683 мс).

**Уровень 3 — Легковесные роутеры** (быстрая одиночная диспетчеризация, масштабирование O(n)):
- **AltoRouter** — быстрый на малых масштабах, но линейный O(n) матчинг приводит к резкому замедлению на масштабе. При 1 000 маршрутах: 201.372 мс (в 295 раз медленнее Waypoint).
- **Bramus** — аналогичный паттерн. Быстрая одиночная диспетчеризация, но O(n) дорого обходится: 268.851 мс при 1 000 маршрутах (в 394 раза медленнее Waypoint).

**Уровень 4 — Некэшированный средний класс**:
- **PHRoute** — построен на ядре FastRoute, но без кэширования, поэтому платит полную стоимость `simpleDispatcher()` за каждый запрос.
- **Nette** — без кэширования, O(n) матчинг.

**Уровень 5 — Тяжёлая инициализация**:
- **League** — PSR-15, без кэширования.
- **Laravel** — самая тяжёлая инициализация. Включает полную инфраструктуру маршрутизации фреймворка (Container, Dispatcher, Events).

### Ключевые выводы

#### 1. Waypoint занимает 1-е место в 4 из 9 сценариев

Кэшированный prefix-trie Waypoint обеспечивает субмиллисекундную диспетчеризацию во всех сценариях — **1-е место в 4 сценариях, 2-е в 4 и 3-е в 1** (динамический первый маршрут, где Symfony занимает 2-е):

| Сценарий | FastRoute | **Waypoint** | 3-е место | Waypoint vs FastRoute |
|----------|-----------|-------------|-----------|----------------------|
| 100 стат., первый | 0.001 мс | **0.002 мс** | Symfony 0.004 мс | 2.0× |
| **100 стат., последний** | 0.002 мс | **0.002 мс (1-е)** | AltoRouter 0.015 мс | **1.0× (ничья)** |
| 100 дин., первый | 0.001 мс | **0.004 мс (3-е)** | Symfony 0.002 мс | 4.0× |
| **100 дин., последний** | 0.002 мс | **0.002 мс (1-е)** | Symfony 0.009 мс | **1.0× (ничья)** |
| 100 стат., все | 0.012 мс | **0.031 мс** | Symfony 0.634 мс | 2.6× |
| 100 дин., все | 0.090 мс | **0.093 мс** | Symfony 0.484 мс | 1.03× |
| **500 смеш., все** | 0.437 мс | **0.322 мс (1-е)** | Symfony 10.023 мс | **0.74× (Waypoint побеждает!)** |
| 100x50 повтор | 4.460 мс | **5.008 мс** | Symfony 24.498 мс | 1.12× |
| **1000 смеш., все** | 1.547 мс | **0.683 мс (1-е)** | Symfony 39.172 мс | **0.44× (Waypoint побеждает!)** |

Разрыв между Waypoint и FastRoute сокращается по мере роста количества маршрутов, и **Waypoint обгоняет FastRoute начиная с 500 маршрутов**. При 1 000 смешанных маршрутах Waypoint быстрее в 2.3 раза.

#### 2. При 500+ маршрутах Waypoint обгоняет FastRoute

Точка пересечения, где prefix-trie Waypoint становится быстрее скомпилированных regex FastRoute, находится на уровне ~500 маршрутов:

- **500 смешанных маршрутов**: Waypoint 0.322 мс (1-е) vs FastRoute 0.437 мс (2-е, на 36% медленнее)
- **1 000 смешанных маршрутов**: Waypoint 0.683 мс (1-е) vs FastRoute 1.547 мс (2-е, на 127% медленнее)

Это доказывает, что структура данных prefix-trie масштабируется значительно лучше скомпилированных регулярных выражений при большом количестве маршрутов. Regex-группы FastRoute растут линейно с количеством маршрутов, тогда как trie Waypoint обеспечивает поиск за время, близкое к константному.

#### 3. Прогрев раскрывает архитектуру кэширования

Разделение прогрева и запроса наглядно показывает, какие роутеры выигрывают от кэширования и как работает их загрузка кэша:

| Роутер | Прогрев (стат. первый) | Запрос (стат. первый) | Коэффициент прогрев/запрос |
|--------|----------------------|---------------------|--------------------------|
| **Waypoint** | 1.662 мс | 0.002 мс | **831×** |
| FastRoute | 0.005 мс | 0.001 мс | 5× |
| Symfony | 0.045 мс | 0.004 мс | 11× |
| AltoRouter | 0.991 мс | 0.012 мс | 83× |
| Laravel | 35.401 мс | 1.015 мс | 35× |

Waypoint показывает наибольшее соотношение прогрев/запрос (831×) — первый запрос несёт значительные накладные расходы на загрузку кэша, но последующие запросы выполняются молниеносно. У FastRoute прогрев почти идентичен запросу, поскольку его скомпилированный кэш — это простой `require` PHP-массива. Для некэшируемых роутеров время прогрева и запроса практически совпадает — они платят одну и ту же стоимость инициализации каждый раз.

#### 4. Waypoint значительно превосходит Symfony на масштабе

`CompiledUrlMatcher` Symfony конкурентоспособен при одиночных диспетчеризациях (2-е место в динамическом первом маршруте — 0.002 мс). Но в сценариях с множественными запросами Waypoint быстрее на порядок:

| Сценарий | Waypoint | Symfony | Преимущество Waypoint |
|----------|----------|---------|----------------------|
| 100 стат., все | 0.031 мс | 0.634 мс | **в 20× быстрее** |
| 100 дин., все | 0.093 мс | 0.484 мс | **в 5.2× быстрее** |
| 500 смеш., все | 0.322 мс | 10.023 мс | **в 31× быстрее** |
| 100x50 повтор | 5.008 мс | 24.498 мс | **в 4.9× быстрее** |
| 1000 смеш., все | 0.683 мс | 39.172 мс | **в 57× быстрее** |

При 1 000 маршрутах Waypoint **в 57 раз быстрее** Symfony. Это демонстрирует, что кэшированный prefix-trie Waypoint фундаментально эффективнее для повторных диспетчеризаций, чем скомпилированный URL-матчер Symfony.

#### 5. Waypoint превосходит все легковесные роутеры на масштабе

Структура данных prefix-trie даёт Waypoint огромное структурное преимущество над O(n) матчерами (AltoRouter, Bramus). Легковесные роутеры конкурентоспособны в одиночных диспетчеризациях, но не могут угнаться за Waypoint на масштабе:

| Сценарий | Waypoint | AltoRouter | Bramus |
|----------|----------|------------|--------|
| 100 стат., все | **0.031 мс (2-е)** | 0.875 мс (4-е) | 2.352 мс (5-е) |
| 100 дин., все | **0.093 мс (2-е)** | 4.721 мс (5-е) | 2.624 мс (4-е) |
| 500 смеш., все | **0.322 мс (1-е!)** | 49.252 мс (4-е) | 73.770 мс (5-е) |
| 100x50 повтор | **5.008 мс (2-е)** | 236.271 мс (5-е) | 131.460 мс (4-е) |
| 1000 смеш., все | **0.683 мс (1-е!)** | 201.372 мс (4-е) | 268.851 мс (5-е) |

При 1 000 маршрутах Waypoint **в 295 раз быстрее** AltoRouter и **в 394 раза быстрее** Bramus. Время диспетчеризации O(n) матчеров растёт линейно с количеством маршрутов, тогда как trie-кэш Waypoint обеспечивает поиск за время, близкое к константному.

#### 6. Waypoint vs League: trie-матчер vs PSR-7 пайплайн

League — единственный роутер в бенчмарке, который создаёт реальные PSR-7 объекты Request/Response при каждой диспетчеризации. Waypoint, как и FastRoute и Symfony, использует свой матчер напрямую и возвращает простые строки. Эта архитектурная разница вносит свой вклад в разрыв:

| Сценарий | Waypoint | League | Разрыв |
|----------|----------|--------|--------|
| 100 стат., первый | 0.002 мс | 0.161 мс | в 81× быстрее |
| 100 дин., все | 0.093 мс | 18.586 мс | в 200× быстрее |
| 500 смеш., все | 0.322 мс | 374.844 мс | в 1 164× быстрее |
| 1000 смеш., все | 0.683 мс | 1 492.054 мс | **в 2 184× быстрее** |

Разрыв частично объясняется накладными расходами PSR-7 у League (создание `ServerRequest`/`Response` при каждой диспетчеризации) и частично отсутствием кэширования — League переинициализируется и перерегистрирует все маршруты при каждом запросе.

#### 7. Диспетчеризация последнего динамического маршрута: Waypoint наравне с FastRoute

В диспетчеризации последнего динамического маршрута (худший сценарий для матчинга) Waypoint занимает **1-е место** с результатом 0.002 мс — наравне с FastRoute (0.002 мс) и значительно опережая Symfony (0.009 мс), Bramus (0.037 мс) и AltoRouter (0.083 мс). Это подтверждает, что prefix-trie обеспечивает стабильно быстрый поиск независимо от позиции маршрута, в отличие от O(n) матчеров, где производительность деградирует для последних маршрутов.

#### 8. Laravel стабильно последний

Полная инфраструктура маршрутизации Laravel делает его самым тяжёлым роутером. Один запрос при 100 статических маршрутах занимает 1.015 мс (время запроса) — в 1 015 раз медленнее FastRoute. При 1 000 маршрутах полный цикл диспетчеризации занимает 3 246 мс против 0.683 мс у Waypoint. Это ожидаемо — роутер Laravel проектировался для полнофреймворочного контекста, где стоимость инициализации амортизируется в течение жизненного цикла запроса.

### Рекомендации

| Сценарий использования | Рекомендуемый роутер |
|----------------------|---------------------|
| Максимальная производительность при ≤100 маршрутах | **FastRoute** (с `cachedDispatcher`) |
| Лучшая производительность на масштабе (500+ маршрутов) | **Waypoint** (с кэшем) — обгоняет FastRoute начиная с 500 маршрутов |
| Производительность, близкая к FastRoute, с trie-матчингом | **Waypoint** (с кэшем) — в пределах 1–4× при ≤100 маршрутах, быстрее при 500+ |
| Фреймворк со скомпилированным матчингом | **Symfony Routing** (с `CompiledUrlMatcher`) |
| Маленькое приложение, мало маршрутов (<50), без кэширования | **AltoRouter** или **Bramus** |
| PSR-15 стек с полным пайплайном | **League Route** |
| Ядро FastRoute без кэширования | **PHRoute** |

### Об этом бенчмарке

- Каждый запрос имитирует реальный жизненный цикл PHP-FPM: роутер инициализируется с нуля или загружается из кэша, затем диспетчеризирует
- Кэшируемые роутеры (FastRoute, Symfony, Waypoint) прогревают кэш до цикла замеров — как при реальном деплое
- **Прогревочный запрос** выполняется первым для отделения накладных расходов холодного кэша (opcache, автозагрузчик и т.д.) от стабильной производительности
- **Прогрев (мс)** = время первого запроса (холодный); **Запрос (мс)** = медиана последующих запусков (тёплый)
- Некэшируемые роутеры платят полную стоимость `initialize() + registerRoutes()` при каждом запросе — это реальная цена отсутствия кэша
- Все роутеры обрабатывают идентичные наборы маршрутов, сгенерированные детерминированно через `RouteGenerator`
- Результаты получены за **20 запусков (медиана)** 2026-02-22, PHP 8.4.5, Linux
- Все роутеры используют свои базовые движки матчинга напрямую и возвращают строки, кроме **League**, который диспетчеризирует через полный PSR-7 пайплайн (создавая `ServerRequest`/`Response` при каждом запросе) — это отражает стандартный паттерн использования League

## Архитектура

Каждый роутер обёрнут в адаптер, реализующий `AdapterInterface`:

```php
interface AdapterInterface
{
    public function getName(): string;
    public function initialize(): void;
    public function registerRoutes(array $routes): void;
    public function dispatch(string $method, string $uri): string;
}
```

Роутеры с поддержкой кэширования также реализуют `CacheableAdapterInterface`:

```php
interface CacheableAdapterInterface extends AdapterInterface
{
    public function warmCache(array $routes, string $cacheDir): void;
    public function initializeFromCache(string $cacheDir): void;
    public function clearCache(string $cacheDir): void;
}
```

Бенчмарк автоматически определяет кэшируемые адаптеры и использует оптимальный путь:
- **Кэшируемые**: `warmCache()` один раз → каждый запрос вызывает `initializeFromCache()` + `dispatch()`
- **Некэшируемые**: каждый запрос вызывает `initialize()` + `registerRoutes()` + `dispatch()`

Маршруты генерируются детерминированно через `RouteGenerator` для обеспечения честного сравнения.
