# Бенчмарк роутера Waypoint

Набор тестов для сравнения роутера [Waypoint](https://github.com/ascetic-soft/Waypoint) с популярными PHP-библиотеками маршрутизации.

## Тестируемые роутеры

| Роутер | Пакет | Версия | Описание |
|--------|-------|--------|----------|
| **Waypoint** | [ascetic-soft/waypoint](https://github.com/ascetic-soft/Waypoint) | v1.2.1 | PSR-15 роутер с prefix-trie матчингом |
| **FastRoute** | [nikic/fast-route](https://github.com/nikic/FastRoute) | v1.3.0 | Популярный standalone regex-роутер |
| **Symfony** | [symfony/routing](https://symfony.com/doc/current/routing.html) | v7.4.4 | Компонент маршрутизации Symfony |
| **Laravel** | [illuminate/routing](https://github.com/illuminate/routing) | v11.47.0 | Роутер фреймворка Laravel |
| **League** | [league/route](https://github.com/thephpleague/route) | v6.2.0 | PSR-15 совместимый роутер |
| **Nette** | [nette/routing](https://github.com/nette/routing) | v3.1.2 | Роутер фреймворка Nette |
| **Bramus** | [bramus/router](https://github.com/bramus/router) | v1.6.1 | Легковесный regex-роутер |

## Требования

- PHP >= 8.4
- ext-mbstring
- Composer

## Установка

```bash
git clone https://github.com/ascetic-soft/Waypoint-benchmark.git
cd Waypoint-benchmark
composer install
```

## Использование

### CLI-бенчмарк (быстрое сравнение)

Запустить все сценарии со всеми роутерами:

```bash
php bin/benchmark
```

Фильтр по роутеру:

```bash
php bin/benchmark --router=waypoint,fastroute,symfony
```

Фильтр по сценарию:

```bash
php bin/benchmark --scenario=static,dynamic
```

Доступные сценарии: `static`, `dynamic`, `highload`

Изменить количество запусков (по умолчанию 20):

```bash
php bin/benchmark --runs=50
```

### PHPBench (статистические бенчмарки)

Запустить все бенчмарки:

```bash
vendor/bin/phpbench run --report=aggregate
```

Фильтр по группе:

```bash
vendor/bin/phpbench run --report=aggregate --group=static
vendor/bin/phpbench run --report=aggregate --group=dynamic
vendor/bin/phpbench run --report=aggregate --group=highload
```

### Makefile

```bash
make install      # composer install
make bench        # Запуск CLI-бенчмарка
make phpbench     # Запуск PHPBench
make all          # Установка + бенчмарк
```

## Методология

> **Каждый бенчмарк измеряет реальный цикл запрос-ответ с раздельным замером прогрева и рабочего запроса.**
>
> Каждый «запрос» = полная инициализация роутера + диспетчеризация:
> - **Кэшируемые роутеры** (FastRoute, Symfony, Waypoint): кэш прогревается один раз, затем каждый запрос загружает данные из кэша + диспетчеризирует
> - **Некэшируемые роутеры**: каждый запрос инициализирует роутер с нуля + регистрирует все маршруты + диспетчеризирует
>
> Замер времени разделён на две фазы:
> - **Прогрев (Warmup)** — первый (холодный) запрос после построения кэша. Может включать промахи opcache, накладные расходы автозагрузчика и т.д.
> - **Запрос (Request)** — медиана последующих (тёплых) запросов. Отражает стабильную производительность в продакшене.
>
> Все 9 роутеров соревнуются в каждом сценарии на равных условиях. Роутеры без поддержки кэширования платят полную стоимость инициализации — это их компромисс.

## Анализ и выводы

### Классификация по уровням

На основе стабильного времени запроса (после прогрева), где кэшируемые роутеры используют свой кэш:

**Уровень 1 — Самая быстрая обработка запросов** (скомпилированные/кэшированные диспетчеры):
- **FastRoute** — самый быстрый в 8 из 9 сценариев. `cachedDispatcher()` загружает скомпилированные regex-данные из PHP-файла, что делает время прогрева и рабочего запроса практически одинаковым. 0.012 мс на запрос при 100 статических маршрутах.
- **Waypoint** — матчинг на основе prefix-trie с поддержкой кэша. **Стабильно 2-е место в каждом сценарии и 1-е при 1 000 смешанных маршрутах** (20.7 мс против 25.5 мс у FastRoute). После загрузки кэша (прогрев) последующие запросы выполняются менее чем за миллисекунду для одиночных диспетчеризаций. В данном бенчмарке Waypoint использует свой trie-матчер напрямую (без PSR-15 middleware-пайплайна), так же как FastRoute и Symfony используют свои базовые матчеры.

**Уровень 2 — Кэшированный средний класс** (хорош для одиночных запросов, отстаёт на масштабе):
- **Symfony** — `CompiledUrlMatcher` хорошо показывает себя при одиночных диспетчеризациях (3-е место в динамическом первом маршруте — 0.056 мс), но уходит на далёкое 3-е место во всех сценариях с множественными запросами. При 1 000 маршрутах в 161 раз медленнее Waypoint (3 344 мс против 20.7 мс).

**Уровень 3 — Легковесные роутеры** (быстрая одиночная диспетчеризация, масштабирование O(n)):
- **AltoRouter** — быстрый на малых масштабах, но линейный O(n) матчинг приводит к резкому замедлению на масштабе. При 1 000 маршрутах: 5 682 мс (в 274 раза медленнее Waypoint).
- **Bramus** — аналогичный паттерн. Быстрая одиночная диспетчеризация, но O(n) дорого обходится: 7 081 мс при 1 000 маршрутах (в 342 раза медленнее Waypoint).

**Уровень 4 — Некэшированный средний класс**:
- **PHRoute** — построен на ядре FastRoute, но без кэширования, поэтому платит полную стоимость `simpleDispatcher()` за каждый запрос.
- **Nette** — без кэширования, O(n) матчинг.

**Уровень 5 — Тяжёлая инициализация**:
- **League** — PSR-15, без кэширования.
- **Laravel** — самая тяжёлая инициализация. Включает полную инфраструктуру маршрутизации фреймворка (Container, Dispatcher, Events).

### Ключевые выводы

#### 1. Waypoint занимает 2-е место в каждом сценарии (1-е при 1 000 маршрутах)

Благодаря разделению прогрева и рабочего запроса видно, что кэшированный prefix-trie Waypoint обеспечивает субмиллисекундную диспетчеризацию во всех сценариях с одиночным запросом — **стабильно 2-е место**, сразу после FastRoute:

| Сценарий | FastRoute | **Waypoint** | 3-е место | Waypoint vs FastRoute |
|----------|-----------|-------------|-----------|----------------------|
| 100 стат., первый | 0.012 мс | **0.021 мс** | Symfony 0.063 мс | 1.8× |
| 100 стат., последний | 0.011 мс | **0.023 мс** | AltoRouter 0.810 мс | 2.1× |
| 100 дин., первый | 0.017 мс | **0.033 мс** | Symfony 0.056 мс | 1.9× |
| 100 дин., последний | 0.028 мс | **0.036 мс** | Symfony 0.710 мс | 1.3× |
| 100 стат., все | 0.615 мс | **1.548 мс** | Symfony 38.916 мс | 2.5× |
| 100 дин., все | 1.612 мс | **2.763 мс** | Symfony 37.716 мс | 1.7× |
| 500 смеш., все | 8.201 мс | **10.410 мс** | Symfony 841.313 мс | 1.3× |
| 100x50 повтор | 80.884 мс | **137.371 мс** | Symfony 1 850.490 мс | 1.7× |
| **1000 смеш., все** | 25.469 мс | **20.707 мс** | Symfony 3 344.249 мс | **0.8× (Waypoint побеждает!)** |

Разрыв между Waypoint и FastRoute сокращается по мере роста количества маршрутов: от 2.5× при 100 статических маршрутах до **победы Waypoint** при 1 000 смешанных маршрутах.

#### 2. При 1 000 маршрутах Waypoint обгоняет FastRoute

При 1 000 смешанных маршрутах prefix-trie Waypoint становится быстрее скомпилированных regex FastRoute:

- **Waypoint**: 20.707 мс (1-е место)
- **FastRoute**: 25.469 мс (2-е место, на 23% медленнее)

Это доказывает, что структура данных prefix-trie масштабируется лучше скомпилированных регулярных выражений при большом количестве маршрутов. Regex-группы FastRoute растут линейно с количеством маршрутов, тогда как trie Waypoint обеспечивает поиск за время, близкое к константному.

#### 3. Прогрев раскрывает архитектуру кэширования

Разделение прогрева и запроса наглядно показывает, какие роутеры выигрывают от кэширования и как работает их загрузка кэша:

| Роутер | Прогрев (стат. первый) | Запрос (стат. первый) | Коэффициент прогрев/запрос |
|--------|----------------------|---------------------|--------------------------|
| **Waypoint** | 1.872 мс | 0.021 мс | **89×** |
| FastRoute | 0.015 мс | 0.012 мс | 1.3× |
| Symfony | 0.093 мс | 0.063 мс | 1.5× |
| AltoRouter | 0.899 мс | 0.354 мс | 2.5× |
| Laravel | 41.002 мс | 12.095 мс | 3.4× |

Waypoint показывает наибольшее соотношение прогрев/запрос (89×) — первый запрос несёт значительные накладные расходы на загрузку кэша, но последующие запросы выполняются молниеносно. У FastRoute прогрев почти идентичен запросу, поскольку его скомпилированный кэш — это простой `require` PHP-массива. Для некэшируемых роутеров время прогрева и запроса практически совпадает — они платят одну и ту же стоимость инициализации каждый раз.

#### 4. Waypoint значительно превосходит Symfony на масштабе

`CompiledUrlMatcher` Symfony уверенно занимает 3-е место при одиночных диспетчеризациях (0.056–0.819 мс). Но в сценариях с множественными запросами Waypoint быстрее на порядок:

| Сценарий | Waypoint | Symfony | Преимущество Waypoint |
|----------|----------|---------|----------------------|
| 100 стат., все | 1.548 мс | 38.916 мс | **в 25.1× быстрее** |
| 100 дин., все | 2.763 мс | 37.716 мс | **в 13.7× быстрее** |
| 500 смеш., все | 10.410 мс | 841.313 мс | **в 80.8× быстрее** |
| 100x50 повтор | 137.371 мс | 1 850.490 мс | **в 13.5× быстрее** |
| 1000 смеш., все | 20.707 мс | 3 344.249 мс | **в 161.5× быстрее** |

При 1 000 маршрутах Waypoint **в 161 раз быстрее** Symfony. Это демонстрирует, что кэшированный prefix-trie Waypoint фундаментально эффективнее для повторных диспетчеризаций, чем скомпилированный URL-матчер Symfony.

#### 5. Waypoint превосходит все легковесные роутеры на масштабе

Структура данных prefix-trie даёт Waypoint огромное структурное преимущество над O(n) матчерами (AltoRouter, Bramus). Легковесные роутеры конкурентоспособны в одиночных диспетчеризациях, но не могут угнаться за Waypoint на масштабе:

| Сценарий | Waypoint | AltoRouter | Bramus |
|----------|----------|------------|--------|
| 100 стат., все | **1.548 мс (2-е)** | 49.633 мс (4-е) | 73.711 мс (5-е) |
| 100 дин., все | **2.763 мс (2-е)** | 85.310 мс (5-е) | 74.103 мс (4-е) |
| 500 смеш., все | **10.410 мс (2-е)** | 1 434.145 мс (4-е) | 1 784.132 мс (5-е) |
| 100x50 повтор | **137.371 мс (2-е)** | 4 299.406 мс (5-е) | 3 699.565 мс (4-е) |
| 1000 смеш., все | **20.707 мс (1-е!)** | 5 682.079 мс (4-е) | 7 081.422 мс (5-е) |

При 1 000 маршрутах Waypoint **в 274 раза быстрее** AltoRouter и **в 342 раза быстрее** Bramus. Время диспетчеризации O(n) матчеров растёт линейно с количеством маршрутов, тогда как trie-кэш Waypoint обеспечивает поиск за время, близкое к константному.

#### 6. Waypoint vs League: trie-матчер vs PSR-7 пайплайн

League — единственный роутер в бенчмарке, который создаёт реальные PSR-7 объекты Request/Response при каждой диспетчеризации. Waypoint, как и FastRoute и Symfony, использует свой матчер напрямую и возвращает простые строки. Эта архитектурная разница вносит свой вклад в разрыв:

| Сценарий | Waypoint | League | Разрыв |
|----------|----------|--------|--------|
| 100 стат., первый | 0.021 мс | 6.117 мс | в 291× быстрее |
| 100 дин., все | 2.763 мс | 737.192 мс | в 267× быстрее |
| 500 смеш., все | 10.410 мс | 15 403.482 мс | в 1 480× быстрее |
| 1000 смеш., все | 20.707 мс | 61 783.328 мс | **в 2 983× быстрее** |

Разрыв частично объясняется накладными расходами PSR-7 у League (создание `ServerRequest`/`Response` при каждой диспетчеризации) и частично отсутствием кэширования — League переинициализируется и перерегистрирует все маршруты при каждом запросе.

#### 7. Диспетчеризация последнего динамического маршрута: Waypoint почти не уступает FastRoute

В диспетчеризации последнего динамического маршрута (худший сценарий для матчинга) Waypoint занимает **2-е место** с результатом 0.036 мс — всего на 25% позади FastRoute (0.028 мс) и значительно опережая Symfony (0.710 мс), Bramus (0.996 мс) и AltoRouter (1.380 мс). Это подтверждает, что prefix-trie обеспечивает стабильно быстрый поиск независимо от позиции маршрута, в отличие от O(n) матчеров, где производительность деградирует для последних маршрутов.

#### 8. Laravel стабильно последний

Полная инфраструктура маршрутизации Laravel делает его самым тяжёлым роутером. Один запрос при 100 статических маршрутах занимает 12.095 мс (время запроса) — в 1 008 раз медленнее FastRoute. При 1 000 маршрутах полный цикл диспетчеризации занимает 140.8 секунд против 20.7 мс у Waypoint. Это ожидаемо — роутер Laravel проектировался для полнофреймворочного контекста, где стоимость инициализации амортизируется в течение жизненного цикла запроса.

### Рекомендации

| Сценарий использования | Рекомендуемый роутер |
|----------------------|---------------------|
| Максимальная производительность при ≤500 маршрутах | **FastRoute** (с `cachedDispatcher`) |
| Лучшая производительность на большом масштабе (1 000+ маршрутов) | **Waypoint** (с кэшем) — обгоняет FastRoute при 1 000 маршрутах |
| Производительность, близкая к FastRoute, с trie-матчингом | **Waypoint** (с кэшем) — всего в 1.3–2.5× медленнее при ≤100 маршрутах |
| Фреймворк со скомпилированным матчингом | **Symfony Routing** (с `CompiledUrlMatcher`) |
| Маленькое приложение, мало маршрутов (<50), без кэширования | **AltoRouter** или **Bramus** |
| PSR-15 стек с полным пайплайном | **League Route** |
| Ядро FastRoute без кэширования | **PHRoute** |

### Об этом бенчмарке

- Каждый запрос имитирует реальный жизненный цикл PHP-FPM: роутер инициализируется с нуля или загружается из кэша, затем диспетчеризирует
- Кэшируемые роутеры (FastRoute, Symfony, Waypoint) прогревают кэш до цикла замеров — как при реальном деплое
- **Прогревочный запрос** выполняется первым для отделения накладных расходов холодного кэша (opcache, автозагрузчик и т.д.) от стабильной производительности
- **Прогрев (мс)** = время первого запроса (холодный); **Запрос (мс)** = медиана последующих запусков (тёплый)
- Некэшируемые роутеры платят полную стоимость `initialize() + registerRoutes()` при каждом запросе — это реальная цена отсутствия кэша
- Все роутеры обрабатывают идентичные наборы маршрутов, сгенерированные детерминированно через `RouteGenerator`
- Результаты получены за **один запуск** 2026-02-18, PHP 8.4.5, Linux
- Все роутеры используют свои базовые движки матчинга напрямую и возвращают строки, кроме **League**, который диспетчеризирует через полный PSR-7 пайплайн (создавая `ServerRequest`/`Response` при каждом запросе) — это отражает стандартный паттерн использования League

## Тестовые сценарии

### 1. Диспетчеризация статических маршрутов
100 статических маршрутов. Каждый «запрос» инициализирует роутер (из кэша или с нуля) и диспетчеризирует один маршрут. Тестирует лучший случай, худший случай и полный перебор.

### 2. Диспетчеризация динамических маршрутов
100 динамических маршрутов (с параметрами `{id}`, `{slug}`). Тот же цикл запроса. Тестирует извлечение параметров и regex-матчинг в реалистичных условиях.

### 3. Высокая нагрузка / Большой масштаб
- 500 смешанных маршрутов: 500 запросов (инициализация + диспетчеризация каждого маршрута)
- 100 динамических маршрутов x50 повторов: 5 000 запросов
- 1 000 смешанных маршрутов: 1 000 запросов (инициализация + диспетчеризация каждого маршрута)

## Архитектура

Каждый роутер обёрнут в адаптер, реализующий `AdapterInterface`:

```php
interface AdapterInterface
{
    public function getName(): string;
    public function initialize(): void;
    public function registerRoutes(array $routes): void;
    public function dispatch(string $method, string $uri): string;
}
```

Роутеры с поддержкой кэширования также реализуют `CacheableAdapterInterface`:

```php
interface CacheableAdapterInterface extends AdapterInterface
{
    public function warmCache(array $routes, string $cacheDir): void;
    public function initializeFromCache(string $cacheDir): void;
    public function clearCache(string $cacheDir): void;
}
```

Бенчмарк автоматически определяет кэшируемые адаптеры и использует оптимальный путь:
- **Кэшируемые**: `warmCache()` один раз → каждый запрос вызывает `initializeFromCache()` + `dispatch()`
- **Некэшируемые**: каждый запрос вызывает `initialize()` + `registerRoutes()` + `dispatch()`

Маршруты генерируются детерминированно через `RouteGenerator` для обеспечения честного сравнения.
